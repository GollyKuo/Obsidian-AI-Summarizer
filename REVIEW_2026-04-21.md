# Media Summarizer 檢視紀錄

日期：2026-04-21

本文件記錄此次 code review 中發現的問題，以及本輪已套用的修正。

## 已修正問題

1. 錯誤的下載恢復邏輯可能會重用不相關的舊檔案。
   - 先前行為：當 YouTube 下載失敗時，恢復流程會掃描整個 `downloads/` 目錄，並挑出最大的檔案。
   - 風險：程式可能在看似成功的情況下，實際摘要了錯誤的媒體內容。
   - 修正：現在每次下載都使用獨立的 session 目錄，恢復流程只會考慮該次 session 建立的檔案。

2. 網頁摘要功能的依賴不完整，在乾淨環境中可能導致程式無法啟動。
   - 先前行為：`src/scraper.py` 在模組載入時就直接 import `trafilatura`，但 `requirements.txt` 中沒有宣告這個套件。
   - 風險：只執行 `pip install -r requirements.txt` 並不足以啟動程式，GUI 甚至可能在開啟前就失敗。
   - 修正：已將缺少的套件加入 `requirements.txt`，並把網頁擷取改成只有在使用該功能時才 lazy-load `trafilatura`。

3. `gui_app.py` 中的 `finish_processing` 被定義了兩次。
   - 先前行為：後面的定義會靜默覆蓋前面的版本。
   - 風險：UI 收尾行為不一致，且後續維護容易變得脆弱。
   - 修正：已將處理收尾邏輯整併成單一方法。

4. Obsidian 輸出在標題相同時可能覆蓋既有筆記。
   - 先前行為：筆記路徑固定為 `<title>.md`，並以寫入模式直接覆蓋。
   - 風險：靜默資料遺失。
   - 修正：現在輸出路徑會自動避開重名，必要時附加 ` (2)`、` (3)` 等後綴。

5. 取消行為只有部分生效，通常要等到下一個階段邊界才會停止。
   - 先前行為：停止按鈕只會設一個旗標，但長時間執行的下載與 AI 呼叫不一定會持續檢查它。
   - 風險：使用者已經取消後，程式仍持續執行，造成不佳體驗與額外資源浪費。
   - 修正：已加入協作式取消 helper，套用到下載與處理流程，並把清理收斂到 `finally`，也不再把取消當成一般錯誤處理。

## 剩餘說明

- Gemini SDK 的請求一旦已經送出，未必能被強制中止。本次修正已讓 app 流程中的取消反應更即時，並阻止後續步驟繼續執行，但已經開始的 SDK 呼叫仍可能在背景中完成。
- 專案目前仍然沒有自動化測試套件。本次驗證主要依賴靜態檢查與編譯。
